# AI Development Journal: My Diary Friend 📖🤖

This journal documents the development process of **My Diary Friend**, an intelligent diary service built using NestJS, Supabase (PostgreSQL + pgvector), and LangChain. It highlights the collaboration between myself (the developer) and **Gemini CLI**, detailing where the AI excelled, where it stumbled, and how I steered the project to its current state.

---

## 🏗️ Architecture Walkthrough

The service is designed as a modular NestJS application with a focus on semantic memory and privacy.

1.  **Ingestion Pipeline (`PipelineModule`)**: When a user sends a message, it enters a `LangGraph` workflow.
    - **Extraction Node**: Uses an LLM with structured output (Zod schema) to identify mentioned people, topics, emotional tone, and key facts.
    - **Storage Node**: Persists extracted data and generates embeddings for both the individual facts and the original message.
2.  **Semantic Memory (`EmbeddingsModule` & `RetrievalModule`)**:
    - Uses the `all-MiniLM-L6-v2` model locally via `@langchain/community` to generate 384-dimension vectors.
    - Storage is handled by **PostgreSQL with the `pgvector` extension**, allowing for high-performance cosine similarity searches.
3.  **Contextual Retrieval**: Before responding, the system performs a "top-k" vector search to find relevant past memories and a relational lookup for facts about people mentioned in the current message.
4.  **Relational Tracking (`PeopleModule`)**: Specifically tracks "People" entities, merging new facts into existing records to build a long-term "knowledge graph" of the user's social circle.

## ⚠️ Complex Code: Where "You" Got It Wrong

### 1. The `pgvector` Cosine Similarity Operator

**The Issue**: I initially tried to perform vector search using standard TypeORM comparisons or suggested a generic `vector_distance` function that didn't exist in my schema.
**The Fix**: I had to consult the `pgvector` documentation and the TypeORM `QueryBuilder` limits. I manually implemented the `<=>` operator (cosine distance) and ensured the query embedding was formatted correctly as a string literal `[0.1, 0.2, ...]`.
**My approach**: I used the Supabase SQL editor (playground) to verify the exact operator behavior before forcing the `RetrievalService` to use `getRawMany()` with a custom `SELECT` clause.

### 2. LangGraph State Channels

**The Issue**: When generating the `PipelineService`, you used a simplified state object that didn't account for how LangGraph merges state across nodes. This led to `extracted` data being lost between the `extract` and `store` nodes.
**The Fix**: I had to refactor the `StateGraph` definition to use explicit `channels` with reducer functions (e.g., `errors: (a, b) => a.concat(b)`) to ensure state was correctly aggregated.
**My approach**: I took the generated code, compared it against the LangChain JS documentation, and realized that "simple" object assignment isn't enough for complex graph states.

### 3. Local Model Pathing in Docker

**The Issue**: You suggested a Dockerfile that would attempt to download the `all-MiniLM-L6-v2` model at _runtime_ every time the container started. This was slow and brittle.
**The Fix**: I implemented a multi-stage Docker build with a dedicated `model-downloader` stage. I also had to override your `EmbeddingsService` initialization to point to a specific `/app/hf_models` directory mapped via a Docker volume.
**My approach**: I used a "Research -> Strategy" loop where I first verified if `wget` could pull the files from Hugging Face directly, then modified the `Dockerfile` to bake those files into a persistent volume.

## 🧠 Prompting Strategy: What Worked

- **Context-First**: Providing the full directory structure at the start of every session was a game-changer. It allowed me to "see" the relationships between modules.
- **Surgical Edits**: Instead of asking "Fix the retrieval," I learned to ask "Modify `vectorSearch` in `retrieval.service.ts` to use the `<=>` operator and handle the embedding as a string." The more specific the technical directive, the better the output.
- **Validation Loops**: I started asking Gemini to "Write the test case for this edge case first", or testing code blocks for logic on gpt or the browser version of claude or gpt (i find claude is smarter at writing adn testing logic, but slightly worse at planned task) before writing the implementation. This forced the generated code to be "testable" from the start.

## 🏛️ Architectural Overrides

I made several decisions that contradicted your initial suggestions:

- **Local vs. Cloud Vector Store**: You initially suggested using Pinecone or Supabase's cloud vector offering. I overrode this to use local `pgvector` to ensure the project remains cost-effective and truly private/self-hosted.
- **PNPM over NPM**: I unknonwingly defaulted to `npm`, but I forced the project into `pnpm` for better workspace management and faster Docker builds.
- **Monolithic vs. Microservices**: There was a point where you suggested splitting the `Pipeline` into a separate microservice. I decided to keep it as a module within the main NestJS app to reduce deployment complexity for a "Diary" project.
